# Release Channels

This document records how `kano-agent-backlog-skill` release publication should
map to branches, tags, and external repositories for the native C++ release
line.

## Source of truth

`main` is the source branch for code, version files, release documentation,
GitHub workflow definitions, and package-manager recipe templates that live in
this repository.

Create versioned releases from an annotated tag on `main`, for example:

```bash
git tag -a v0.0.4 -m "Release 0.0.4"
git push origin v0.0.4
```

Jenkins and GitHub Actions must build from a pushed source revision or the final
tag. Local-only changes cannot prove release readiness on remote builders.

## GitHub Release

GitHub Releases are tag based. They should be created from `v0.0.4` after the
native release artifacts are produced and checksummed.

Attach platform archives, checksum files, and any manifest sidecars generated by
the build pipeline. Do not use an orphan branch for GitHub Release assets.

Completion gate:

- The public [releases page](https://github.com/kanohorizonia/kano-agent-backlog-skill/releases)
  must show the versioned release.
- The release must include downloadable Windows, Linux, and macOS artifacts for
  the built platforms from the matching Jenkins `Build_CI` source/version.
- The release must include integrity metadata such as the checksum/index files
  generated by Jenkins.
- Dry-run `Release_Publish` output is review evidence only; it does not satisfy
  the release completion gate.

## GitHub Pages

GitHub Pages is branch based in this repository. The source docs live on
`main`, while the generated site is published to `gh-pages` by
`.github/workflows/pages.yml` and the scripts under `src/shell/docs/`.

The `gh-pages` branch is derived output. Treat it like an orphan-style deploy
branch, not as the source of release metadata.

The public site must expose the same release download and installation path as
the repository README:

- link to the latest GitHub Release and the current version tag;
- describe manual archive installation for Windows, Linux, and macOS;
- describe winget, Homebrew, and apt channel status without claiming that a
  package-manager channel is live before its metadata is actually published.

## Jenkins

Jenkins should consume committed and pushed source changes. A release build must
archive root-level release artifacts, checksum files, and manifest sidecars so
GitHub Releases and package-manager preparation can consume the same build
outputs.

The release proof is the Jenkins build log and archived artifacts from the
pushed revision or tag, not the local workstation build alone.

For `0.0.4`, `Release_Publish` must consume the fresh `Build_CI` artifact set for
the same version. A release that copies stale upstream artifacts from another
version is not acceptable even if the release staging job itself succeeds.

## winget

The current winget path is preparation and validation only. It must not open a
PR against `microsoft/winget-pkgs` unless a human explicitly changes the release
policy for that run.

Prepare the manifest after the GitHub Release assets are final because winget
needs stable download URLs and SHA256 values. Keep repo-local winget templates
and generators on `main`; publish only to the configured owned/internal review
target while `PUBLISH_WINGET` remains a dry-run validation flag.

## Homebrew

Homebrew does not use this repository's `gh-pages` branch.

The first supported path is an owned Kano tap or disposable validation clone.
Do not touch `homebrew-core` and do not open an external Homebrew PR from this
repo. Update a shared tap branch only after explicit approval and after release
assets are final.

## apt

apt publication should go to the configured apt package repository, which owns
the `.deb` files plus `pool/` and `dists/` index metadata.

That repository may be a separate Git repository, object storage bucket, or
other package-hosting target. Do not publish apt metadata to this repo's
`gh-pages` branch unless the project explicitly decides to host a static apt
repository there.

## Jenkins package-manager preparation

The `Release_Publish` Jenkins job is configured to generate package-manager
review payloads with these repo-local scripts:

- winget: `src/shell/release/02-generate-winget-skeleton.sh`
- Homebrew: `src/shell/release/03-generate-homebrew-skeleton.sh`
- apt: `src/shell/release/06-generate-apt-skeleton.sh`

Package-manager publish flags remain dry-run by default. A non-dry-run publish
requires explicit operator flags plus configured owned/internal repository
targets and credentials.

## 0.0.4 status

For `0.0.4`, this repo is prepared for native release metadata and channel
guidance. Actual package-manager publication still requires final release asset
URLs, SHA256 values, and any external tap/package-repository credentials.
