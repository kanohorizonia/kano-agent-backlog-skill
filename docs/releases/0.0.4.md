# Kano Agent Backlog Skill 0.0.4

`0.0.4` is the first Kano Agent Backlog Skill release line where the supported
runtime is the native C++ `kano-backlog` executable. The repo-local `kob` and
`kano-backlog` launchers resolve to the native binary and no longer fall back to
Python package entrypoints.

The old `0.0.3` Python-public planning line was not tagged and is superseded by
this native release.

## Highlights

- Native C++ CLI is the supported executable surface.
- Python package entrypoints, editable Python installs, pytest oracle tests,
  and PyPI/TestPyPI publishing are retired for this milestone.
- Native release gates cover build, smoke tests, runtime contract checks,
  command coverage, GitHub Actions cloud builds, and release-phase validation.
- Public docs and GitHub Pages target the native `0.0.4` release line.

## Download and Install

Use the platform asset from the GitHub Release download list.

- Windows: install the `.msi` when present. It installs the skill to
  `%USERPROFILE%\.agents\skills\kano-agent-backlog-skill` and adds the installed
  `scripts` directory to the per-user PATH so `kob` can be launched from any
  directory.
- macOS: use Homebrew from an owned Kano tap once the tap formula is published.
  Until then, use the macOS `.tar.gz` archive as the portable install path.
- Linux: use the Linux `.tar.gz` archive for this release. apt metadata is still
  a dry-run package-manager lane until `.deb` authoring and repository signing
  are enabled.

After install, run:

```bash
kob version
kob doctor
```

## Package Manager Status

- winget: dry-run/internal manifest preparation only. This release must not open
  a public `microsoft/winget-pkgs` PR unless that is explicitly approved.
- Homebrew: validate through an owned Kano tap or disposable local tap clone.
  Do not touch `homebrew-core`.
- apt: dry-run recipe preparation only. Debian package authoring and signed apt
  repository metadata are not enabled for this release.

## Release gate expectations

Expected local gates before tagging:

- `pixi run build-dev`
- `pixi run quick-test`
- `bash src/shell/test/native-command-coverage.sh --strict`
- `pixi run native-runtime-gate`
- `bash src/shell/test/lint.sh`
- `pixi run build-release`
- `bash scripts/kob --version`
- `bash scripts/kob doctor`

Jenkins and GitHub Actions must build from a pushed source revision or tag so
their logs prove the same version that will be released.

## Distribution preparation

- GitHub Release: tag `v0.0.4` from `main`, create the release from that tag,
  and attach only user-facing installable or portable artifacts. Jenkins keeps
  manifest CSV, manifest summary JSON, checksums, release indexes, report
  payloads, and package-manager recipe payloads as internal review evidence.
- GitHub Pages: docs are generated from `main` and published through the
  `gh-pages` deploy branch. The public site must link the GitHub Release
  download page and describe manual archive installation plus package-manager
  channel status.
- winget: prepare manifests only after GitHub Release asset URLs and SHA256
  values are stable. The current release rehearsal is dry-run/internal only and
  must not open a `microsoft/winget-pkgs` PR.
- Homebrew: validate through an owned Kano tap or disposable local tap clone.
  Do not touch `homebrew-core`; update a shared tap branch only after explicit
  approval.
- apt: publish `.deb` packages and repository metadata to the configured apt
  package repository. Do not use this repo's `gh-pages` branch for apt unless a
  static apt repository is deliberately designed there.
- Jenkins `Release_Publish` generates package-manager review payloads, but
  package-manager publish remains off by default.

## Known limitations

- Package-manager manifests are preparation work until final release asset URLs,
  checksums, and repository targets exist.
- If `v0.0.4` is absent from GitHub Releases or has no downloadable platform
  artifacts, the 0.0.4 release remains incomplete regardless of dry-run Jenkins
  release evidence.
- Optional exact tokenizer or embedding providers must be native adapters before
  they can enter the supported executable contract.
